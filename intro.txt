第 1 段 · Setup：图流是什么、为什么需要 sketch
图流——由带时间戳的边构成的交互序列——支撑着许多现代数据系统：金融交易网络、社交平台、流行病接触追踪、骨干路由遥测等。由于这些工作负载在数量级上超出了精确存储与精确查询的预算，过去十年的研究发展出了一系列基于 sketch 的图流摘要（graph stream summarization）结构，以有界的误差换取常数时间更新和亚线性内存。

这一段的任务：交代研究领域、让读者知道我们在谈"sketch-based graph stream summarization"这个具体家族。

第 2 段 · 核心引入：倾斜性 + 关键变量 $A_v$
现实世界中的每一个图流都具备一个共同特征——度数偏斜（degree skew）。出度分布和边频分布都是重尾的，少数 hub 节点占据了大部分活动量。传统的均匀分配 sketch（如 TCM、GSS）给每个节点相同的矩阵空间，结果 hub 的数据严重冲突，而长尾节点的格子长期闲置。

最近出现的一批 skew-aware sketch——Scube、SIM、Auxo、HIGGS——通过一个共同的设计原语来解决这个失衡：

$$
A_v = f(\hat{d}_v)
$$

其中 $\hat{d}_v$ 是对节点 $v$ 度数的在线估计，$f$ 是一个单调、确定性的映射函数。每个节点的内存占用和查询代价都随其度数伸缩；hub 获得额外的行或子矩阵，长尾节点保持最小预算。

这个原语既是 skew-aware sketch 效率优势的根源，也是本文指出的一个至今未被研究的隐私漏洞的根源。

这一段的任务：把论文的靶心变量 $A_v$ 用公式 (1) 明确点名。后面所有讨论都绕着这个变量展开。

第 3 段 · 核心洞察：三条泄漏通道（论文的"心脏"）
由于 $A_v$ 随 $\hat{d}_v$ 单调增长，而 $A_v$ 又直接决定了每次节点查询要扫描多少行矩阵，任何黑盒的查询延迟观察者都可以还原节点间的度数排序，而完全不需要读取任何 cell 的内容。更一般地说，我们识别出三条黑盒通道，攻击者可以通过它们破坏 skew-aware sketch 中的 hub 隐私：

(1) 值通道（Value Channel）
攻击者按照 nodeWeightQuery 返回的值排序节点，即可还原真实的 hub 集合。在 stackoverflow 和 wiki-talk 数据集上，Precision@50 = 1.00——返回的前 50 名和真实前 50 hub 完全一致。

(2) 时间通道（Timing Channel）
攻击者记录每次查询的延迟，利用 $\text{cost}(v) \propto A_v \propto \hat{d}_v$ 来推断度数。实测 Pearson 相关系数：

Scube：$r \geq 0.9$
TCM、GSS：$r = 0$（因为它们的每节点代价是常数）
(3) 边探测通道（Edge-Probing Channel）
攻击者对可疑的边调用 edgeWeightQuery；指纹匹配使误报率低至 $2^{-\ell}$（$\ell$ 位指纹），允许以 AUC ≈ 1.0 的精度进行成员推断。

其中通道 (1) 和 (3) 是均匀分配 sketch 所共有的，属于经典 DP 范围内的问题；而通道 (2)——时间通道——在任何均匀分配 sketch 中都不存在，也从未在以往的图流隐私研究中被分析过。 它是 $A_v$ 这个结构原语的直接后果：去掉 $A_v$ 通道就消失，但同时 skew-aware 的效率优势也消失。解决这一矛盾是本文的核心贡献。

这一段的任务：这是全篇最关键的段落。它完成了三件事——命名三条通道、给出具体实验数字、明确指出时间通道是 skew-aware 范式独有的新问题。

第 4 段 · 现实意义：这个问题为什么值得解决
图流越来越多地用于建模高度敏感的交互：

接触追踪：腾讯健康码在疫情期间处理了超过 240 亿次扫码，涉及十亿用户。在这种接触图流中，识别 top-K hub 就等于识别潜在"超级传播者"的身份和轨迹。
金融风控：图流 sketch 用于识别欺诈团伙，其中的 hub 节点对应机密的 VIP 商户或洗钱账户。
网络安全：骨干路由遥测会向能够探测 sketch 的对手暴露僵尸网络 C&C 层级的拓扑。
在每一个场景中，需要被保护的不是单条边的内容，而是重点节点本身的「结构身份」、「交互频率」和「时序访问签名」。

这一段的任务：用一个真实例子（健康码）把抽象的攻击具象化，让审稿人相信这不是一个"纸上威胁"。

第 5 段 · 差异化：为什么传统 DP 还不够
差分隐私（DP）是敏感流数据抗推断发布的标准形式化框架，已有一些 DP 版本的频率 sketch（DPSW、Count-Min 隐私分析、LDP heavy hitter）成功部署。但直接把它套用在 skew-aware 图流摘要上会遇到两个障碍：

障碍 1——重尾数据下的单一预算困境
在重尾数据上，单一 $\varepsilon$ 预算无法同时做到：

掩盖 hub（信号大 → 需要大噪声）
保留长尾（信号接近噪底 → 不能加大噪声）
噪声调到前者，后者被淹没；调到后者，前者泄漏。

障碍 2（更根本）——DP 管不到查询延迟
DP 的不可区分性保证是在"算法输出分布"上定义的。查询延迟不是输出，不在 DP 的保护范围内。

无论输出层加多少扰动，都无法关闭时间通道，因为 $A_v$ 本身没有被随机化，循环次数不是一个 DP 保护的可观测量。已有的 heavy-hitter-aware DP 工作只解决障碍 1，而且只在频率维度里解决，它们不触及 $A_v$。

这一段的任务：明确建立论文的"gap"——为什么现有方法不够。为下一段提出 P-Scube 做好铺垫。

第 6 段 · 解决方案：扩展 DP 的观测面
我们提出 P-Scube——一个隐私保护型的 skew-aware 摘要架构，在形式化的 $(\varepsilon, \delta)$-DP 框架内关闭全部三条通道。

核心思想：扩大 DP 的观测面，把内部分配变量 $A_v$ 也纳入保护对象。

具体做法：

在度数检测器的估计值中注入校准过的 Laplace 噪声，使 $A_v$ 成为一个 $\varepsilon_A$-DP 的输出
由 DP 的后处理不变性，$A_v$ 诱导的查询延迟 $T_v \propto A_v$ 继承同样的保证，时间通道在结构上被关闭
在此基础上，对值通道和边通道的输出应用非对称、个性化的 DP 预算——给 hub 更严格的隐私保护，给长尾更宽松的预算，使得信号最脆弱的地方（长尾）效用损失最小
这一段的任务：用一句话讲清楚"P-Scube 到底做了什么"——扩展 DP 的观测面，这是论文的核心定位。

第 7 段 · 研究挑战
实现这一设计需要解决三个非平凡的挑战：

C1｜隐私感知的 heavy-hitter 检测：注入到在线度数检测器的 DP 噪声不能淹没检测信号，而检测器本身要在严格的常数内存约束下工作（每个 slot 一个概率计数器）。
C2｜无差别（oblivious）自适应分配：随机化 $A_v$ 必须保留 Scube 在长尾工作负载上的效率优势，同时让真实度数相同的节点，其 $A_v$ 分配在统计上不可区分。
C3｜非对称的效用-隐私分配：每节点的 $\varepsilon_v$ 调度必须：对 hub 足够强以实现真正的 hub 隐私；对长尾足够温和以保留长尾效用；并合成为全局的 $(\varepsilon, \delta)$ 保证。
这一段的任务：把解决方案分解成三个具体子问题，给读者一个"这篇论文要做哪些硬骨头"的全景。

第 8 段 · 贡献
我们的贡献包括三方面：

① 首个对 skew-aware 图流 sketch 的系统性隐私分析
我们在两个真实重尾数据集上形式化并实证验证了三条黑盒泄漏通道（值 / 时间 / 边探测）。时间通道在均匀分配 sketch 中不存在，它是 Scube、SIM、Auxo、HIGGS 所共享的 $A_v$ 原语的结构性后果。据我们所知，这是首个把 $A_v$ 视作"攻击者的一级可观测量"的隐私威胁刻画。

② P-Scube：DP 观测面扩展的 skew-aware 摘要架构
我们设计了一个三层机制：

层 1：随机化 $A_v$ 分配（防时间通道）
层 2：节点权重查询的个性化 DP 预算（防值通道）
层 3：边查询的频率自适应随机响应（防边通道）
并证明其合成在扩展的观测元组 $(\tilde{W}v, A_v', \tilde{e}{s,d})$ 上满足 $(\varepsilon, \delta)$-DP。

③ 理论界 + 实验验证
我们推导出分析性的界，把时间通道相关性 $|r(d_v, T_v)|$ 与 $\varepsilon_A$、长尾 ARE 与 $\varepsilon_v(d_v)$ 直接联系起来。在 stackoverflow 和 wiki-talk 上：

时间通道相关性：$r \approx 0.95 \to r < 0.3$
Hub Precision@50：$1.00 \to < 0.3$
长尾 ARE 代价：低于 5%
